# 琴行约课小程序从零到一开发教程

> 状态：持续更新  
> 项目：琴行学生约课与老师授课排班小程序  
> 维护方式：后续开发过程中，每完成一个阶段，同步补充本教程中的开发记录、关键决策、实现步骤与验证结果。

## 1. 教程目标

这份教程用于记录一个琴行约课小程序从需求梳理到最终跑通的完整过程。它既是项目开发日志，也是后续可复用的小程序开发教程。

教程会覆盖：

- 如何从业务资料整理产品需求。
- 如何拆分学生端、老师端、管理端和服务端能力。
- 如何设计数据模型、预约规则、课时规则和权限规则。
- 如何搭建小程序工程、后台服务和本地开发环境。
- 如何逐步实现登录、约课、改课、课表、合同、上课记录等核心功能。
- 如何调试、测试、部署，并让小程序完整跑通。

## 2. 当前项目背景

当前项目已完成需求草案沉淀，核心业务来自琴行约课场景：

- 已签约学生或家长可以通过小程序查看合同、个人信息、校区地址，并预约或修改课程。
- 琴行老师可以查看今日课表、学生信息、门店地图、收款码和上课记录。
- 管理员需要维护校区、老师、学生、合同、课程、课时规则、排课与预约数据，并能清理或停用已过期学员、已离职老师。
- 正式上线后需要支持学生、老师、后台管理员三类权限；小程序前端不单独提供管理员入口，只保留一个前端巡检账号，可按入口切换查看学生端和老师端运行状态。
- 第一版需要支持多校区，当前已有两个分店。
- 当前两个教学地点均在长春市：一店为净月区御翠园，二店为高新区融创上城二期，均位于对应小区的别墅区内。
- 当前课程品类包括架子鼓、吉他、钢琴。
- 老师默认周二至周日每天 12:00-20:00 全时段排满，每节课 45 分钟；每周一全店固定休息。
- 学生端任一时刻只展示一个约课日期：20:00 前只能预约当天，20:00 后自动切换到第二天。
- 老师请假可由后台按天或按具体时段关闭对应约课时间；体验课由后台创建，不走学生端自助约课。
- 学生距离开课时间大于或等于 1.5 小时时可以改课。
- 小程序需要考虑微信转发分享能力，但分享入口必须受角色和页面权限限制，不能通过分享绕过登录和数据权限。
- 后续计划接入老师考勤管理，可选择与钉钉打卡联动，或在小程序内实现轻量考勤；课时统计以小程序上课记录为准，考勤以打卡或签到记录为准，两者通过老师、门店、日期和课节时间形成核对。

详细需求见：[琴行约课小程序需求文档](../requirements/product-requirements-draft.md)。

### 阶段三：技术方案与工程初始化

目标是确定技术栈并搭建可运行的项目骨架。

当前已完成内容：

- 小程序开发方式：原生微信小程序，使用 WXML、WXSS、JavaScript。
- 后端技术栈：Node.js + Express。
- 第一版持久化：JSON 文件存储，默认路径 `backend/data/spinach-music.json`。
- 选择 JSON 文件存储的原因：当前开发环境缺少 `make`，原生 SQLite 依赖无法编译；JSON 存储零原生依赖，更适合先把业务闭环跑通。
- 目录结构已初始化：`miniprogram/` 和 `backend/`。
- 小程序品牌素材已复制到 `miniprogram/assets/brand/`。
- 本地开发端口默认使用 `3010`。

后续可升级内容：

- 服务器准备好后，可继续使用文件存储试运行，也可替换为 SQLite、MySQL 或 PostgreSQL。
- 正式上线前建议增加进程管理、日志、备份和 HTTPS 反向代理。
- 正式微信登录需要接入小程序 AppID、AppSecret 和 `wx.login` code 换取 OpenID。

服务器初步选型与当前状态：

- 早期曾按阿里云轻量应用服务器估算，当前实际已购买腾讯云轻量应用服务器，后端部署步骤以腾讯云 Ubuntu 服务器为准。
- 当前业务属于低到中等并发的小程序后端，访问高峰主要集中在学生约课、改课和老师查看课表时段。
- 以本市 4 家线下培训机构估算，早期更可能是几百名活跃学生、每天几十到数百次预约相关操作，2 核 2G 足够支撑第一版上线和试运营。
- 不建议选择 0.5 GiB 或 1 GiB 内存规格，后端服务、数据库、Nginx、日志和系统进程同时运行时余量太小。
- 若后端、数据库、管理端都放在同一台机器，且预算允许，可以选择 2 vCPU、4 GiB 内存、50 GiB 系统盘作为更稳妥的生产起步配置。
- 4 vCPU、8 GiB 及以上规格暂不建议一开始购买，等监控数据证明 CPU、内存或连接数长期接近瓶颈后再升级。
- 业务图片、合同附件等文件不建议长期存放在服务器本地磁盘，后续应优先考虑对象存储，服务器磁盘主要用于系统、应用、数据库和日志。
- 中国内地服务器对外提供服务需要提前准备域名、ICP备案和 HTTPS 证书。
- 当前营业执照、小程序主体和正式域名尚未完成，因此可以先部署后端并用公网 IP 调试；正式小程序上线前再补齐域名、备案、HTTPS 和微信小程序后台合法域名配置。

### 阶段九：考勤联动与规模化运营

目标是为门店、老师和学生数量增长后的运营管理提前留好扩展空间。

建议分阶段实现：

- 第一阶段在小程序内完成课程记录和有效课时统计，老师每节课后提交上课记录，小程序按自然月汇总有效课时。
- 第二阶段接入钉钉考勤，只把钉钉作为老师到岗、离岗、请假等考勤来源，小程序仍作为课程记录和课时统计来源。
- 第三阶段建立月度核对表：同一老师同一自然月同时展示小程序有效课时、钉钉出勤天数、请假记录和异常项。
- 第四阶段如门店不想维护两套工具，再评估是否在小程序内做轻量考勤，例如门店定位签到、上课前后签到、请假审批和异常申诉。

当前建议：

- 不建议一开始完全替代钉钉考勤。钉钉在员工考勤、审批、组织架构和手机打卡方面更成熟，小程序先专注课程记录与课时核算。
- 小程序可以先做“课时统计主系统”，钉钉作为“考勤来源系统”，两边通过老师 ID、手机号、钉钉员工 ID、门店、日期、时间段做核对。
- 小程序自建考勤不是不可行，但更适合作为第二阶段或补充能力。若只需要“老师是否按课表到店上课”的轻量核对，可以在小程序中做上课前签到、课后提交记录和定位校验；若要做正式员工考勤、迟到早退、请假审批、异常申诉和月度考核，继续使用钉钉会更稳妥。
- 推荐第一版采用混合方案：小程序内记录“课程出勤事实”，钉钉记录“员工到岗事实”。月底由后台自动核对两类事实，而不是让小程序一开始承担完整人事考勤系统职责。
- 后续如果老师人数和门店数量明显增加，再把 JSON 文件存储迁移到 SQLite、MySQL 或 PostgreSQL，并为学生、老师、门店、预约、上课记录、操作日志建立索引。

### 2026-06-03：服务器选型初步建议

本次目标：

- 根据本市计划开设 4 家线下培训机构、当前已有 2 家营业的规模，选择适合第一版小程序的服务器配置。
- 在访问量暂不明确的情况下，避免一开始购买过高配置，同时保留可升级空间。

前置状态：

- 当前仍处于需求与技术方案准备阶段。
- 业务核心是约课、改课、课表、合同和上课记录，不包含视频、直播、大文件下载等高带宽功能。
- 阿里云购买页中可选轻量应用服务器通用型，包含 2 vCPU/0.5 GiB、2 vCPU/1 GiB、2 vCPU/2 GiB、2 vCPU/4 GiB、4 vCPU/8 GiB、4 vCPU/16 GiB 等规格。

初步估算：

- 4 家门店早期可按几百名活跃学生估算。
- 小程序请求主要是短接口请求，单次数据量小。
- 高峰集中在放课表、约课、改课和老师上课前后。
- 第一版更关键的瓶颈通常不是带宽，而是数据库设计、预约时段占用逻辑、备份和监控。

推荐配置：

- 推荐起步：2 vCPU、2 GiB 内存、40 GiB 系统盘。
- 系统镜像：Ubuntu 24.04。
- 购买时长：如果确定要用于备案和正式开发，可选择 1 年；如果只是试验，也可以先短周期购买。
- 备选稳妥配置：2 vCPU、4 GiB 内存、50 GiB 系统盘，适合后端、数据库和管理端都放在同一台服务器，且希望少折腾内存优化的情况。

关键决策：

- 不选 0.5 GiB 和 1 GiB：内存余量过小，后续运行数据库、服务端、Nginx、日志和安全组件会比较吃紧。
- 暂不选 4 vCPU、8 GiB：当前业务访问量不足以支撑这类成本，后续可根据监控升级。
- 优先选择轻量应用服务器：第一版属于小型 Web 应用和小程序后端，轻量服务器足够起步。
- 保留升级路径：当 CPU、内存、磁盘、数据库连接数或接口响应时间出现持续压力，再升级配置或拆分数据库。

后续验证方式：

- 上线后观察 CPU、内存、磁盘、带宽和接口响应时间。

- 学生管理需要支持按门店、状态、课程、老师、到期时间筛选；后续学生数量增多后必须有搜索、分页和批量停用能力。

1. 先补登录与角色模型：学生、老师、管理员，以及默认演示管理员账号。
